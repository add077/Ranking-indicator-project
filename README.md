# Ranking Indicator Project

## Overview

This project studies the cross-sectional usefulness of technical indicators in crypto markets. The core idea is not to predict returns directly from raw prices, but to estimate, for a given asset, date, indicator, and horizon, whether a signal generated by that indicator is likely to be successful.

The pipeline transforms multi-asset OHLCV data into a long-format supervised learning dataset, then benchmarks several classifiers under a strict rolling-window backtest with embargo. The main output is a probability of success, used to rank indicators within each `date × asset × horizon` decision set.

## Research Objective

For each crypto asset and each date, the project aims to answer a practical question:

**Which technical indicators appear most promising for this asset, at this date, and for this forecast horizon?**

This is framed as a cross-sectional classification problem where each observation corresponds to:

`date × asset × indicator_name × horizon`

and the target is binary.

## Pipeline

The research workflow is organized into six stages:

1. **Data loading**
   Multi-timeframe OHLCV data are loaded and standardized from CSV files.

2. **Technical indicator computation**
   A broad library of indicators is computed from OHLCV inputs, including momentum, trend, volatility, band/channel, and volume-based features.

3. **Signal engineering**
   Raw indicators are converted into standardized signal representations:
   - `signal_direction`
   - `signal_active`
   - `signal_strength`
   - `signal_signed_value`

   The transformation depends on the indicator family:
   - centered oscillators
   - extreme-zone oscillators
   - trend filters
   - volume confirmation indicators
   - Bollinger-type band signals
   - context filters

4. **Target construction**
   Targets are created for multiple forward horizons (`1d`, `4d`, `24d`) using future returns aligned at the signal date. The default specification uses cross-sectionally scaled future returns and produces a binary success label.

5. **Regression dataset assembly**
   The wide panel is reshaped into a long-format model dataset with one row per:
   `date × asset × indicator_name × horizon`

6. **Modeling and backtesting**
   Classifiers are trained and evaluated out of sample using a rolling-window protocol with embargo and horizon-aware purge.

## Final Dataset

The final supervised dataset is built in long format. Each row contains:
- `date`
- `asset`
- `indicator_name`
- `signal_family`
- `horizon`
- `signal_direction`
- `signal_active`
- `signal_strength`
- `signal_signed_value`
- `target`

It also includes contextual market-state variables copied at the `date × asset` level, mainly from the `context_filter` family.

This structure makes it possible to compare indicators directly within each asset-date cross section and across forecast horizons.

## Signal Construction

Signals are not built from a single universal rule. Instead, each indicator is assigned to a family, and each family is transformed according to its economic interpretation.

Examples:
- **Centered oscillators** use sign plus cross-sectional leave-one-out z-score activation.
- **Extreme-zone oscillators** are interpreted relative to their natural range and midpoint.
- **Trend filters** are converted through the abnormality of their recent time-series change.
- **Volume confirmation indicators** use the cross-sectional abnormality of their recent delta.
- **Band/channel indicators** currently use an explicit Bollinger-style distance-to-band logic.
- **Context filters** are not converted into directional signals and remain explanatory state variables.

## Target Definition

The target is binary and horizon-specific.

For a signal observed at date `t`, the pipeline computes the forward return between `t` and `t+h`, where `h` can be `1d`, `4d`, or `24d`. By default, this forward return is scaled by the cross-sectional dispersion of future returns across assets at the same date and horizon.

A signal is labeled successful when its direction matches a sufficiently favorable future move. This construction keeps the target attached to the row at date `t`, ensuring proper temporal alignment between signal observation and subsequent realized outcome.

## Models

### Penalized Logistic Regression

The penalized logistic regression serves as the main linear benchmark. It is trained separately for each horizon, since short- and longer-horizon signal dynamics are not assumed to share the same structure.

Depending on the experiment, the model can use:
- classical standardization through `StandardScaler`, fitted on the training fold only
- or cross-sectional standardization, where numerical features are normalized within each date-level panel

Its most important output is `pred_proba_success`, the estimated probability that a signal will be successful.

### Random Forest Classifier

A Random Forest benchmark is also implemented using the exact same dataset, target definition, rolling split, and ranking logic. This provides a non-linear benchmark that can capture interaction effects and threshold behavior that a linear logistic specification may miss.

## Backtesting Protocol

All model evaluation is performed under a strict rolling-window framework.

Key features:
- fixed-size rolling training window
- rolling test window
- embargo between train and test
- horizon-aware purge to avoid target leakage across blocks
- no random split
- no classical k-fold cross-validation

This design is meant to preserve temporal integrity and produce realistic out-of-sample estimates.

## Main Outputs

The pipeline produces several levels of output:
- fold-level performance metrics
- global out-of-sample metrics
- row-level out-of-sample predictions
- indicator rankings within each `date × asset × horizon` group

The most important predictive output is:
- `pred_proba_success`

This probability is used to rank indicators for a given asset and horizon at each date. In practice, the project is therefore not only a classification pipeline, but also a ranking framework for indicator selection.
